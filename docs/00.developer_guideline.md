# 프로젝트 개요
> 프로젝트의 간략한 소개, 프로젝트의 목표와 배경을 설명합니다. 그리고 주로 어떤 문제를 했는지에 대한 이야기를 제공합니다.
## 프로젝트 소개
본 프로젝트는 젊은 세대를 타겟으로 하는 모바일 최적화 패션 쇼핑몰 웹 애플리케이션입니다. 상품 조회부터 장바구니 관리, 결제 시뮬레이션까지 이어지는 일련의 이커머스 경험을 제공합니다

# 시스템 아키텍처
## 전체 구조도
```
Client[User Client (Browser)]  
    subgraph "Frontend (React)"
        App[App.js (Router)]
        Context[Global State (Context API)]
        Pages[Pages: List, Detail, Cart]
        Components[Components: Header, PaymentModal, CardList]
        
        App --> Context
        Context --> Pages
        Context --> Components
    end

    Client --> App
    Context <-->|Real-time Sync| Firestore
    Context <-->|Auth Check| Auth
 ```

##  컴포넌트 및 데이터 흐름
Router: react-router-dom을 사용하여 페이지 간 이동을 처리하며, URL에 따라 적절한 컴포넌트를 렌더링합니다.

State Management:

CartContext: 장바구니 상품 목록, 수량, 금액 계산 로직 관리.

PaymentContext: 결제 모달 상태, 카드 등록/조회, 결제 프로세스 관리.

# 개발 환경 설정 및 규칙
## 개발 환경 설정 (Setup)
프로젝트 실행을 위해 다음 단계가 필요합니다.

저장소 복제 (Clone)

```
git clone <repository_url>
cd <project_folder>
```

패키지 설치
```
npm install
```

실행
```
npm start
```

## 폴더 구조 (Directory Structure)
```
src/
├── components/   # 재사용 가능한 UI 컴포넌트 (Header, Button, PaymentModal 등)
├── pages/        # 라우팅되는 페이지 단위 (ProductList, CartPage, ProductDetail)
├── contexts/     # 전역 상태 관리 (CartContext, PaymentContext)
├── firebase/     # Firebase 초기화 및 설정 파일
├── assets/       # 이미지, 폰트 등 정적 파일
└── App.js        # 라우팅 및 Provider 설정
```

## 브랜치 전략 (Git Flow)
- main: 배포 가능한 안정화 버전 (Production)
- develop: 개발 중인 최신 버전
- feature/<기능명>: 새로운 기능 개발 시 사용 (예: feature/payment-module)

작업 완료 후 develop 브랜치로 Pull Request(PR) 및 Merge.

## 커밋 메시지 규칙 (Conventional Commits)
- feat: 새로운 기능 추가
- fix: 버그 수정
- docs: 문서 수정
- style: 코드 포맷팅, 세미콜론 누락 등 (로직 변경 없음)
- refactor: 코드 리팩토링

예시: feat: 장바구니 수량 증감 기능 추가


# 프로젝트에서 중요했던 지점
## 보안 중심의 결제 모듈 설계
결정: 사용자 편의를 위해 카드 등록 기능을 제공하되, 보안 리스크를 원천 차단하기 위해 DB에 카드 번호 원본 저장을 금지했습니다.
구현: CardRegistrationForm에서 입력받은 정보 중 last4(마지막 4자리)와 별칭, 브랜드 정보만 저장하고, 나머지는 즉시 폐기하거나 가상 토큰으로 대체하는 로직을 적용했습니다.

## '10만원 이상 무료배송' 정책 로직화
결정: 배송비 계산 로직이 여러 컴포넌트(CartPage, PaymentModal, OrderSummary)에 파편화되는 것을 방지해야 했습니다.
구현: CartContext 내부의 totalPrice 계산 로직에 배송비 정책을 포함하거나, 전용 유틸리티 함수로 분리하여 단일 진실 공급원(Single Source of Truth)을 유지했습니다.

## Fast URL을 통한 빠른 피드백 루프
방향성: 기획/디자인의 잦은 변경에 대응하기 위해 로컬 호스트가 아닌, 외부 접속이 가능한 배포 URL(Vercel/Netlify)을 제공하는 것을 원칙으로 했습니다. 이를 통해 모바일 실기기 테스트를 즉각적으로 수행할 수 있었습니다.

# 다른 문서를 어떻게 참고하면 좋을지에 대한 가이드
## 기존 문서 참고 방법
본 프로젝트는 기능별 상세 명세가 별도의 문서로 관리되고 있습니다. 기능 수정 시 반드시 해당 문서를 먼저 확인하세요.

payments_docs.md: 카드 등록 시퀀스, 보안 정책, 모달 UI 동작 방식 확인 시 참고.

cart_docs.md (장바구니 명세): 수량 증감 로직, 배송비 정책 상세, UI 요구사항 확인 시 참고.

shopping_docs.md: 페이지 간 라우팅 흐름, 상세 페이지 스펙 확인 시 참고.

## 문서 갱신 절차
코드가 변경되어 기존 문서와 불일치가 발생할 경우, 다음 절차를 따릅니다.

Code First: 코드 수정 및 기능 테스트 완료.

Doc Update: 관련 문서(*.md)의 '유의사항 및 알려진 이슈' 또는 '기능 목록' 섹션 업데이트.

Commit: 코드 변경 사항과 문서 변경 사항을 함께 커밋 (feat: ... & docs: 문서 업데이트).
